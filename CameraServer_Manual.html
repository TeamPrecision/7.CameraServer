1. Overview

CameraServer is a Windows web application that keeps a USB or built-in camera open at all times and exposes it through a REST API and an in-browser dashboard. It is designed for automated vision testing on manufacturing or QA lines where the camera must respond quickly to test requests without the delay of opening and closing the device each time.

Key capabilities:

  • Persistent camera — opens once at startup, stays open until manually stopped.
  • Blocking test API — a single POST /api/test/start call waits until the result is ready and returns it in one response.
  • Five test modes — barcode reading, image comparison, LED color detection, LED blink measurement, and live observation.
  • MJPEG live stream and snapshot accessible from any browser on the network.
  • Structured JSONL logging with in-browser log viewer and monthly file export.
  • Auto-recovery — black-frame detection triggers automatic camera reopen.

3. Installation & Setup

3.1 Install .NET 8

Download and install the .NET 8 Runtime (or SDK if you plan to build) from https://dotnet.microsoft.com/download.

3.2 Project Files

Place the CameraServer project folder anywhere on the machine. The working directory when running will be the project root. The application will automatically create:

  • data/settings.json — persisted runtime settings
  • logs/camera_YYYY-MM.jsonl — monthly log file

3.3 Configure appsettings.json

{
  "Urls": "http://0.0.0.0:5000",
  "Camera": {
    "Port": 0,
    "Api": "DShow",
    "TargetFps": 30,
    "BlackFrameThreshold": 5,
    "BlackFrameMean": 8.0,
    "ReopenDelayMs": 800
  }
}
KeyDefaultDescription
Urlshttp://0.0.0.0:5000Listening address. Use 0.0.0.0 to accept connections from all network interfaces.
Camera:Port0Camera device index. 0 = first camera, 1 = second, etc.
Camera:ApiDShowDShow (recommended) or Any.
Camera:TargetFps30Initial capture rate. Overridden by saved settings.
Camera:BlackFrameThreshold5Consecutive black frames before triggering auto-reopen.
Camera:BlackFrameMean8.0Pixel mean value below which a frame is classified as black.
Camera:ReopenDelayMs800Milliseconds to wait before reopen attempt.

4. Running the Server

4.1 Development

cd CameraServer
dotnet run

4.2 Production (pre-built)

dotnet publish -c Release -o publish
cd publish
dotnet CameraServer.dll

4.3 Run as a Windows Service (auto-start)

# Register (run once as Administrator)
sc create CameraServer binPath="dotnet C:\CameraServer\publish\CameraServer.dll" start=auto

# Start
sc start CameraServer

4.4 Access the Dashboard

Open a browser and navigate to:

  • Same machine: http://localhost:5000
  • Remote machine: http://<server-ip>:5000
NoteIf the camera does not appear immediately, wait 2–3 seconds for the status pill to turn green (Ready).

6. Camera Tab

The Camera tab controls the physical camera device and its runtime settings.

6.1 Status Panel

Shows live read-only information updated every 2 seconds:

FieldDescription
StateCurrent camera state (Ready, BlackFrame, Reopening, Stopped, Error).
PortActive device index.
ResolutionActual width × height returned by the driver.
FPSMeasured capture rate (averaged over 2 seconds).
UptimeTime since the server started.
Black framesCount of consecutive black frames detected.

6.2 Start / Stop Camera

⏹ Stop Camera — Disposes the camera handle. The physical camera LED turns off. The capture loop enters idle mode (no CPU usage for frame processing).

▶ Start Camera — Reopens the camera using the current port and resolution settings. Only visible when the camera is stopped.

🔄 Reopen — Force-disposes and reopens the camera without changing any settings. Useful to recover from a temporary hardware issue.

WarningStopping the camera while a test is running will cause the test to time out.

6.3 Port Selection

Enter the camera device index (integer, starting from 0) and click Set Port. The server immediately disposes the current camera and reopens it on the new port. The new port is persisted automatically.

ValueMeaning
0First camera (default, usually the built-in webcam)
1Second camera (first external USB camera)
2, 3, …Additional cameras in enumeration order

6.4 Resolution

Select a preset from the dropdown or choose Custom… to enter width and height manually, then click Apply.

PresetUse case
Auto (camera default)Lets the driver choose (typically 640×480)
320 × 240Low bandwidth, fast processing
640 × 480Standard definition
1280 × 720HD — recommended for barcode reading
1920 × 1080Full HD — high detail, higher CPU usage
Custom…Any width/height supported by the camera driver
NoteChanging resolution triggers a camera reopen (briefly turns the LED off and on). The new resolution is persisted automatically.

6.5 Rotation

Select the rotation angle from the dropdown. Rotation is applied server-side on every captured frame — it affects the live stream, snapshots, and all test modes including ROI coordinates and barcode decoding.

OptionUse case
0°Normal orientation
90° CWCamera physically mounted rotated left
180°Camera physically mounted upside-down
270° CCWCamera physically mounted rotated right

Rotation takes effect immediately but is not persisted until you click Save Settings.

6.6 Target FPS

Controls how many frames per second the capture loop processes. Lower values reduce CPU and network usage; higher values give faster test response times at the cost of more resources.

Select from 5 / 10 / 15 / 20 / 25 / 30 / 60 fps. Takes effect immediately but is not persisted until you click Save Settings.

TipFor barcode reading, 15–20 fps is usually sufficient. For BlinkLed at 2 Hz, use at least 20 fps (Nyquist: at least 2× the blink frequency).

6.7 Save Settings

Click 💾 Save Settings to write the current port, resolution, FPS, and rotation to data/settings.json. These values will be restored automatically on next server start, overriding appsettings.json.

Port and resolution changes are auto-persisted when applied (since they require a reopen). FPS and rotation changes require an explicit Save.

6.8 Camera Properties Dialog

Click ⚙️ Open Camera Properties to open the camera manufacturer's native settings window on the server screen. This dialog is provided by the camera driver and typically contains:

  • Exposure (auto/manual)
  • White balance
  • Focus (auto/manual)
  • Anti-flicker (50 Hz / 60 Hz)
  • Brightness, contrast, saturation, sharpness

The capture loop pauses while the dialog is open. The stream resumes automatically when the dialog is closed.

WarningThe dialog opens on the physical monitor connected to the server, not in the browser. You must have access to the server screen to use this feature.

7. Test Tab

The Test tab is the primary interface for running vision tests. A test request is sent to POST /api/test/start which blocks until the result is ready (pass/fail) or the timeout expires.

7.1 Common Fields

FieldDescription
HeadIdentifier for the test head / station (e.g. A, B, Station1). Included in logs and results.
StepName of the test step (e.g. read2d, check_led). Included in logs and results.
ModeTest algorithm to run (see sections below).
Timeout (ms)Maximum time to wait for a result before returning Timeout status. Default: 30 000 ms.
ROI X / Y / W / HRegion of Interest in pixels. Set all to 0 to use the full frame. The ROI rectangle is drawn in red on the live stream during a test.

All fields are remembered in browser local storage and restored on page reload.

7.2 NORMAL Normal Mode

No image processing is performed. The camera keeps streaming and the test waits until the timeout expires, then returns TIMEOUT. Useful for keeping the camera active during observation steps in automated sequences.

7.3 READ2D Read2d Mode

Reads a 2D DataMatrix barcode from the camera frame using ZXing.Net.

ParameterDefaultDescription
Expected digits10Required barcode string length. Set to 0 to accept any length.
Pre-rotate °0Rotate the ROI by this many degrees before attempting to decode. Useful when the barcode is printed at a fixed angle.

Algorithm: The server tries the frame at 0° then rotates ±5° increments up to ±90° until a barcode is decoded or the timeout expires.

Result: PASS — barcode decoded with correct digit count. FAIL — decoded but wrong digit count. TIMEOUT — not decoded within the timeout.

7.4 COMPARE Compare Mode

Compares the live frame against one or more reference images using KAZE feature matching.

ParameterDefaultDescription
Reference folderconfig/{step}Folder containing reference images. Defaults to config/{step name}.
Steps (image count)1Number of sequential reference images to match (1 for single-image comparison).

Reference image naming: {Head}Image1.png, {Head}Image2.png, …

Example: Head = A, Steps = 2 → needs AImage1.png and AImage2.png in the reference folder.

Result: PASS when all steps are matched in sequence. FAIL if a reference file is missing or no match is found within the timeout.

7.5 CHECKLED CheckLed Mode

Detects whether a solid-color LED is on and stays on for a minimum duration.

ParameterDefaultDescription
Color spaceBGRBGR (Blue-Green-Red, 0–255 each) or HSV (H: 0–179, S/V: 0–255).
Low [0] [1] [2]0, 0, 0Lower bound of the color range (channel 0, 1, 2).
High [0] [1] [2]255, 255, 255Upper bound of the color range.
Min pixels100Minimum number of pixels in range to consider the LED "on".
Hold ms500The LED must remain on continuously for this many milliseconds.
Tip — Finding HSV valuesUse a color picker tool to get the RGB value of the LED, then convert to HSV. For a green LED (RGB 0, 255, 0), the HSV range is approximately H: 55–65, S: 200–255, V: 100–255. HSV is more robust to lighting variation than BGR.

Result: PASS when the color holds for the required duration. TIMEOUT if the LED is never detected.

7.6 BLINKLED BlinkLed Mode

Measures the blink frequency (Hz) and duty cycle (%) of a blinking LED.

ParameterDefaultDescription
Color space, Low, High, Min pixels—Same as CheckLed — defines what counts as "LED on".
Freq Hz0Expected blink frequency. Set to 0 to skip frequency check.
±Tolerance (freq)0.5Acceptable deviation from expected frequency in Hz.
Duty %0Expected duty cycle (0–100%). Set to 0 to skip duty check.
±Tolerance (duty)5Acceptable deviation from expected duty cycle in %.
Sample cycles20Number of full on/off cycles to average before evaluating. Higher = more accurate but slower.
Note — FPS requirementThe capture FPS must be at least 4× the blink frequency for accurate measurement. For a 5 Hz LED, use at least 20 fps.

Result: PASS when both enabled checks are within tolerance. FAIL if frequency or duty is out of range. TIMEOUT if the LED never blinks or cycles cannot be measured.

7.7 Result Display

After a test completes, the result box shows:

FieldDescription
BadgePASS (green), FAIL (red), TIMEOUT (red), ERROR (red), RUNNING (yellow), CANCELLED (red)
ValuePrimary result — e.g. decoded barcode string, "Color Detected", measured frequency/duty.
DetailAdditional information — digit count, matched steps, pixel count, tolerance check results.
TimeElapsed time (seconds) and wall-clock timestamp.

After a test completes, the display automatically freezes on the last frame so you can inspect what the camera saw. Click ✕ Resume to return to live view.

8. Snapshot & Freeze

Clicking 📷 Snapshot fetches the current frame from the server and displays it as a frozen static image in the camera panel. The live stream continues running on the server — only the browser display is frozen.

ActionBehavior
📷 SnapshotFreeze display on the current frame.
📷 Snapshot (again)Update the frozen frame to the latest available frame.
✕ ResumeReturn to live MJPEG stream.
▶ Start TestAutomatically resumes live view so you can watch the test run.
Test completesAutomatically freezes on the last test frame.
TipTo save a snapshot image to disk, right-click the frozen image and choose "Save image as…".

11. Settings Persistence

Settings are stored in two places:

Server-side (data/settings.json)

Stores hardware settings that affect the camera on startup:

{
  "Port":      0,
  "Width":     1280,
  "Height":    720,
  "TargetFps": 30,
  "Rotation":  0
}

This file is loaded at server start and overrides the corresponding values in appsettings.json. It is written when:

  • Port is changed (automatic)
  • Resolution is changed (automatic)
  • 💾 Save Settings is clicked (FPS and rotation)
  • POST /api/camera/save is called via API

Browser-side (localStorage)

Stores all Test tab form fields (mode, timeout, ROI, LED color ranges, etc.) per browser. These values are restored automatically on page reload. Camera hardware settings (port, resolution, FPS, rotation) are always loaded from the server.

12. Troubleshooting

Camera stays on "Opening" or shows "Error"

  • Check that the correct port index is set. Try 0, 1, 2 in order.
  • Ensure no other application (Teams, Zoom, OBS) has the camera open.
  • Try clicking 🔄 Reopen.
  • Check the Logs tab for CAM_OPEN_FAIL or CAM_OPEN_EX messages.

Black frames / camera keeps reopening

  • This is normal during low-light conditions. The auto-recovery will reopen the camera.
  • Increase Camera:BlackFrameMean in appsettings.json if false positives are too frequent (default 8.0).
  • Increase Camera:BlackFrameThreshold to require more consecutive black frames before reopening (default 5).

Barcode not detected (Read2d)

  • Ensure the barcode is clearly visible and in focus.
  • Use the ROI to crop to the barcode area — this improves decode speed and accuracy.
  • Try the Pre-rotate ° field if the barcode is at a fixed angle.
  • Try a higher resolution (1280×720 or 1920×1080).
  • Open the Camera Properties dialog to adjust exposure and focus.

BlinkLed frequency reading is inaccurate

  • Ensure FPS is at least 4× the blink frequency.
  • Increase Sample cycles (e.g. 30–50) for a more stable average.
  • Use a tight ROI around the LED to avoid noise from surrounding pixels.

Server crashes when changing resolution or stopping camera

  • This was a known threading issue, fixed in the current version. Port and resolution changes now use signal flags that are consumed safely by the capture thread.

Dashboard not accessible from another machine

  • Verify Urls in appsettings.json is http://0.0.0.0:5000 (not localhost).
  • Allow port 5000 through Windows Firewall: Windows Defender Firewall → Allow an app → Add port 5000.
